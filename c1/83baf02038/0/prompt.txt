Confirmed, and your objection actually runs deeper than the resolver: lockfile_path_for_tool_source routes every subproject's source through monorepo_lockfile_root() (lockfile.rs:1260), so auto-lock and set_platform_info write into the shared root lockfile too. That means "same short+version, different request options" isn't just a merge-time aggregation case — it's a standing state the write path must tolerate. Two corrections to my proposed changes:

1. Resolver: consensus over requests, not Fn(version, platform)
Since several fresh ToolVersions can share (short, version) with different request options, precompute the decision per (version, platform) across all matching requests, and only rekey on unanimity:

// built at the update_lockfiles call site, per short
let mut rekey: HashMap<(String, String), Option<BTreeMap<String, String>>> = HashMap::new();
for tv in &tool_versions {
    let Ok(backend) = tv.request.backend() else { continue };
    for platform_key in platforms_of_existing_legacy_entries(&tv.version) {
        let resolved = Platform::parse(&platform_key).ok().and_then(|p| {
            backend.resolve_lockfile_options(&tv.request, &PlatformTarget::new(p)).ok()
        });
        match rekey.entry((tv.version.clone(), platform_key)) {
            Entry::Vacant(e) => { e.insert(resolved); }
            Entry::Occupied(mut e) => if *e.get() != resolved { e.insert(None); } // disagreement → don't move
        }
    }
}
None (disagreement, parse failure, or resolve error) means the platform's data stays where it is — never moved, never dropped. That's always safe: each request's own writes land under its exact resolved key anyway, so ambiguity degrades to "no migration", not corruption. The consumed-keys bookkeeping for preserve_absent_tool_entries still works — an entry is consumed only if every platform actually moved.

2. The PR's cleanup loop needs the same medicine — this is a live bug in #10999 as written

(Pre-PR code was also broken here — the old adoption would fold B's entry into A's outright — so it's not a regression, but since the PR is touching this exact line, fix it now.)
